## 🔒 ЗОЛОТОЕ ПРАВИЛО ЗМЕЙКИ — НЕРУШИМО

```
НЫРОК → СДВИГ → ВОЗВРАТ → СДВИГ
```

**Любое предложение по navigator.py, которое нарушает этот цикл хотя бы в одном сценарии — ЗАПРЕЩЕНО.**
Маяк — на берегу, 2 шага вправо, перпендикулярно нырку. Возврат — физически на маяк.

## 🔒 ПРАВИЛО МАЯКА — НАИВЫСШИЙ ПРИОРИТЕТ

**RETURNING останавливается ТОЛЬКО на линии маяка. Ничто иное не останавливает возврат.**
- Реки, ручьи, вода на пути возврата — НЕ помеха, бот идёт сквозь них
- Визуальные проверки воды в RETURNING при активном маяке — ЗАПРЕЩЕНЫ
- Любой код, останавливающий RETURNING до линии маяка — ЗАПРЕЩЁН

## 1. OCR времени марша — ЗАПРЕЩЕНО

**Что было:** Читать время марша Картера через pytesseract из скриншота.
**Почему провалилось:** Нестабильно, зависит от языка игры, Carter overlay перекрывает регион y=773. ACCEL_TIME_REGION (y=295) — другой регион, но тоже нестабилен.
**Решение:** Детерминированная формула `T_one_way = T_max / 2^N`. Игра сама защищает от лишних ускорений.

## 15. EMA angle clamp в CoastalSnakeNavigator — ЗАПРЕЩЕНО

**Что было:** `np.clip(diff_raw, -0.2, 0.2)` перед умножением на `coast_ema_alpha`.
**Почему ломает:** Замедляет адаптацию угла в 5×. Бот 25+ шагов идёт в неверном направлении (в воду) пока EMA догоняет реальный угол берега.
**Решение:** `self._coast_angle = (a + alpha * diff_raw) % (2 * np.pi)` — только нормализация, без clamp. `% (2π)` защищает от unbounded float accumulation.

## 21. OCR-проверка масла с порогом 70K — УДАЛЕНА, не восстанавливать

**Что было:** `OIL_STOP_THRESHOLD = 70_000`, `parse_oil()`, `_read_oil()`, проверка `oil < OIL_STOP_THRESHOLD` перед отправкой капитана.
**Почему удалено:** OCR путает суффикс «М» (миллион) с «К» (тысяча) — 6.06М читается как 6.06К = 6060 < 70000 → ложный аварийный стоп. Ретраи не помогают: если шрифт/рендеринг стабильно даёт К вместо М, все N попыток дадут одно и то же. История чтений тоже не спасает: если глюк случился на первом походе — второго не будет.
**Решение:** Игра сама не даст отправить капитана при нулевом масле. `_send_captain` вернёт False → бот перезапустит цикл. Игра — единственный надёжный рубеж защиты.

## 24. Физическое хождение по океану в DIVING для «перепрыгивания» — ЗАПРЕЩЕНО

**Что было:** Идея: пусть бот идёт сквозь воду во время нырка, пока не найдёт сушу на другой стороне.
**Почему запрещено:** Каждый шаг по океану = потраченное время на исследование пустой воды. При max_inland_steps=10 и move_wait=2с → 20 секунд туда + 20 назад = 40 секунд впустую на каждую «пустую» полосу.
**Решение:** Читать миникарту ДО нырка (fwd['land_px']). Если land_px == 0 → сдвинуться вдоль берега, не входя в воду. «Прыжок» — логический (чтение миникарты), не физический (клики по океану).

## 25. Абортировать нырок при is_water в DIVING — ЗАПРЕЩЕНО

**Что было:** Добавить в DIVING: если is_water == True → аварийный выход из нырка.
**Почему запрещено:** Бот исследует изрезанный рельеф — острова, полуострова, заливы. Между берегом и островком может быть канал (2-4 шага воды). Абортирование при воде = бот никогда не исследует землю за каналом, пропускает биржи.
**Решение:** Проверять land_px ПЕРЕД нырком (на миникарте). Если есть хоть что-то (land_px > 0) — нырять и исследовать. Только чистый океан (land_px == 0) → пропустить колонку.

## 26. Фиксировать dive_vec навсегда для «глобального курса» — ЗАПРЕЩЕНО

**Что было:** Идея зафиксировать dive_vec (направление нырка) в начале сессии как константу.
**Почему запрещено:** Бот обходит всю игровую территорию по большому кругу (как обход пиццы). Берег постоянно поворачивает. Если dive_vec зафиксирован на «юго-восток», через четверть круга нырки будут идти параллельно берегу, а не перпендикулярно — полный провал навигации.
**Решение:** dive_vec остаётся динамическим — определяется от угла берега в HOMING каждый цикл. Фиксировать нельзя ничего кроме уже зафиксированного shift_vec (он и так зафиксирован с первого обнаружения берега).

## 27. Убирать blind phase из RETURNING без замены — ЗАПРЕЩЕНО

**Что было:** Удалить blind фазу RETURNING полностью, заменить на is_water check.
**Почему ломает:** Без blind фазы бот останавливается на каждой внутренней реке (is_water=True на середине пути), не доходит до берега.
**Решение:** Blind phase нужна — она игнорирует реки ГЛУБОКО внутри суши. Убирать только вместе с полноценной альтернативой.

## 28. _peek_step в RETURNING как критерий ОСТАНОВКИ — ЗАПРЕЩЕНО

**Что было:** `if mult is None → shift+HOMING` в RETURNING. Логика: None = берег достигнут.
**Почему ломает:** Когда бот УЖЕ в воде (переступил берег), seaward direction видит сушу позади (берег!) → peek возвращает 1.0 (не None) → бот продолжает идти вглубь воды.
**Решение:** Для остановки в RETURNING использовать `_is_at_coast_now()` (или `is_water` из center screen). peek — ТОЛЬКО для масштабирования шага, не для остановки.

## 29. Смешивать step-size и stop-condition в одном механизме — ЗАПРЕЩЕНО

**Что было:** `mult = _peek_step()`, если None → стоп, иначе → шаг с множителем. Один механизм за две несовместимые задачи.
**Почему ломает:** Критерий "нет суши в 3 экранах" не совпадает с критерием "достигли берега". Работает случайно.
**Решение:** Разделить: (1) `_peek_step` → только размер шага; (2) `_is_at_coast_now()` или `is_water` → только остановка.

## 30. Менять семантику _return_steps с click-count на distance — ОПАСНО

**Что было:** `_return_steps = _dive_distance + 3`, декремент на `mult` (float).
**Почему опасно:** Ломает существующие тесты, неинтуитивно, сложно отлаживать. При мультипликаторе 2.0 бот делает вдвое больше дистанции за меньше кликов.
**Решение:** Если нужна симметрия — отдельно считать `_dive_distance` и `_return_distance`, но не путать с `_return_steps` (click counter).

## 31. Нарушать принцип змейки без явного «да» — КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО

**Что было:** В процессе «улучшений» убирали blind phase, меняли механику RETURNING, добавляли is_water стопы — без согласования.
**Почему запрещено:** Принцип змейки (HOMING→DIVING→RETURNING→SHIFT→repeat) — фундамент. Все улучшения (peek, damper, footprint) — надстройка над ним, не замена.
**Решение:** Перед любым изменением DIVING или RETURNING — явное подтверждение от пользователя. Список допустимых изменений: (1) peek для SIZE шага; (2) дополнительные проверки при переходах состояний. Нельзя: менять саму последовательность состояний.

## 32. Pixel-детект красных пикселей в центре мини-карты как детектор маяка — ЗАПРЕЩЕНО

**Что было:** `_is_at_coast_now()` читал BGR-пиксели в 15px зоне центра мини-карты, искал `red > 150 & red > green*2 & red > blue*2` — красные пиксели footprint overlay.
**Почему ломает:** Зеркальная wall (inland-параллельная линия) рисуется при `extra_coast_steps=2` на той же canvas-строке (y), где бот находится после shift перед возвратом. Каждый шаг RETURNING bot проходит через ячейки mirror wall → красные пиксели ВСЕГДА в центре → `_is_at_coast_now()` возвращает True немедленно → бот топается на месте.
**Решение:** `is_at_beacon(tolerance=1)` — canvas-математика специально сравнивает позицию бота с позицией маяка (не всех footprint-ячеек). Walls на другой оси — не вызывают ложных срабатываний.

## 35. Добавлять `is_at_beacon` в `at_coast` (HOMING) — ЗАПРЕЩЕНО

**Что было:** `at_coast = info['is_at_coast'] OR is_at_beacon(tolerance=1)` в HOMING.
**Почему ломает:** `at_coast=True` → ocean-column check → `fwd['land_px']==0` → SHIFT (если бот в воде). Повторяется бесконечно: бот ходит вдоль воды шифтами и теряется.
**Решение:** Маяк как забор — ТОЛЬКО в ветке движения (else), отдельно от coast-detection:
`at_fence = fwd.is_ocean OR is_water OR is_at_beacon(tolerance=1)` → отступить к суше.
`is_at_coast` остаётся единственным триггером нырка.

---

## 34. Добавлять `is_at_beacon` в `at_fence` HOMING — ЗАПРЕЩЕНО

**Что было:** `at_fence = fwd.is_ocean OR is_water OR is_at_beacon(tolerance=1)` в else-ветке HOMING.
**Почему ломает:** После каждого RETURNING→HOMING старый маяк оказывается на манхэттенском расстоянии 1 от новой позиции бота. `is_at_beacon(tolerance=1)` = True немедленно → бот бесконечно осциллирует у берега (шаг к воде → маяк → отступить → шаг к воде → ...), никогда не достигает `is_at_coast`, не начинает нырок.
**Решение:** `at_fence = fwd.is_ocean OR is_water` — только реальные детекторы. Маяк используется ТОЛЬКО в RETURNING как цель возврата, не как забор в HOMING.

## 33. `_return_steps = MAX_STEPS_SAFETY` в RETURNING — ЗАПРЕЩЕНО

**Что было:** При переходе DIVING→RETURNING ставить `_return_steps = MAX_STEPS_SAFETY = 80` вместо `inland_steps`.
**Почему ломает:** Когда маяк не нарисован (колонка-океан), `is_at_coast_now()` никогда не срабатывает. Бот движется `toward_water=True` (в сторону океана) 80 шагов → уходит глубоко в океан.
**Решение:** `_return_steps = inland_steps` — точная симметрия нырка. Маяк — основная остановка; inland_steps — аварийный backstop cap. Изменить НЕЛЬЗЯ.

## 37. Пересчитывать sv в _shift_click() из _inland_vec — ЗАПРЕЩЕНО

## 40. `_is_at_coast_now` только canvas (без визуальной воды) при активном маяке — ЗАПРЕЩЕНО

**Что было:** `_is_at_coast_now()` возвращал только `is_at_beacon(tolerance=1)` — чистая canvas-математика.
**Почему ломает:** Canvas дрейфует от дробных шагов визуальной навигации (`frac = dist/step_px < 1.0`). Накопленный дрейф сдвигает виртуальную позицию бота относительно реальной. Линия маяка не срабатывает → бот проходит маяк и продолжает идти в воду.
**Решение:** `_is_at_coast_now = beacon_line OR is_water` — визуальная вода подстраховывает когда canvas отклонился.

## 41. `pixels_per_step` хардкодом (20) — ЗАПРЕЩЕНО

**Что было:** `pixels_per_step = 20` хардкодом в навигаторе.
**Почему ломает:** Реальный сдвиг миникарты = ~12px на шаг джойстика. Хардкод 20 = ошибка 67%. Маяк рассчитывается как «2 шага × 20px = 40px» от точки нырка, но реально смещается только на 24px → маяк ставится в неверной точке берега → RETURNING останавливается не там.
**Решение:** `pixels_per_step` = измеренное значение `nav_pps` из `gui_config.json` (заполняется `measure_step_pixels()` в `calibration_ui.py`). Передаётся в навигатор при старте.

**Решение:** Возврат к canvas-математике `_is_beyond_beacon_line` (коммит 536772f). Визуальный возврат НЕ реализовывать без: (1) точной калибровки `pixels_per_step` на реальной карте, (2) реального теста на игре, (3) явного «делай» от пользователя.

## 44. `is_water` в RETURNING — ЗАПРЕЩЕНО (AP-37, Хангоф #18)

## 45. Добавлять визуальную воду в `_is_at_coast_now` при активном маяке — ЗАПРЕЩЕНО (AP-38, Хангоф #18)

## 46. «Маяк не найден → движение к воде» — ЗАПРЕЩЕНО (AP-46, Хангоф #19)

**Что было:** В RETURNING_SCAN и RETURNING_BEACON: если beacon_pos=None → `_move_perpendicular(toward_water=True)`. Логика «раз не нашли — идём дальше к берегу».

**ЗОЛОТОЕ ПРАВИЛО BEACON:** Маяк рисуем МЫ — ОБЯЗАНЫ найти. Не найден = СТОЯТЬ на месте, сканировать следующий шаг. Никаких движений к воде без маяка.
**Решение:** `if beacon_pos is None: return True` — стоим и ждём следующего шага. В BEACON → SCAN без движения.

## 50. Два механизма сглаживания угла одновременно — ЗАПРЕЩЕНО (Хангоф #21)

**Что было:** EMA (`coast_ema_alpha=0.3`, хардкод) + угловой демпфер (`max_pitch_delta=15°`) работали одновременно. Двойное ограничение давало ~1/3 реальной манёвренности: бот адаптировался к изгибу берега в 3× медленнее чем нужно.
**Почему запрещено:** Оба решают одну задачу (плавный поворот inland_vec), но суммируются в двойное торможение. Пользователь видит «ватное» управление.
**Решение:** Единый параметр `smooth_alpha` (0.1–1.0) управляет EMA напрямую. Демпфер (`_clamp_vec`, `_prev_inland_vec`) удалён. π-флип защита (diff > π/2) остаётся как жёсткий инвариант — не отключается.
**Рабочая конфигурация:** `smooth_alpha=0.70` — быстрая адаптация, проверено на 30+ локациях.
